def polite_get(url, params=None):
    session.cookies.clear()
    response = session.get(url, params=params, headers=headers)
    response.raise_for_status()
    if not getattr(response, "from_cache", False):
        time.sleep(0.2)
    return response

def resolve_titles(titles):
    rows = []
    params = {
        "action": "query",
        "format": "json",
        "prop": "info",
        "redirects": 1,
        "titles": "|".join(titles)
    }
    data = polite_get(base_url, params).json()

    followed = {}
    for kind in ("normalized", "redirects"):
        for step in data["query"].get(kind, []):
            followed[step["from"]] = step["to"]

    pages = {p["title"]: p for p in data["query"]["pages"].values()}
    for title in titles:
        final = title
        while final in followed:
            final = followed[final]
        rows.append({
            "doc_id": title,
            "resolved": final,
            "page_id": pages.get(final, {}).get("pageid")
        })
    return pl.DataFrame(rows)

Answer: The pinned revision’s timestamp matches latest’s exactly, and the returned wikitext opens with the same infobox and short description add_doc_id would already show for Donald Knuth today, confirming revids= really does hand back one immutable version rather than whatever the live article currently says. formatversion=2 does not change the categorymembers list’s Python type here (both are already lists in this particular response shape); the difference formatversion changes is in how prop="info"’s pages block is keyed, a dict by page ID under the default and a plain list under version 2. The revision ID matters more for reproducibility because it names one fixed version of the data itself: without recording it, a rerun of this notebook next year gets a different Donald Knuth. The format version only fixes the shape of the response envelope; whichever version is requested, the underlying page content is identical. Losing track of a format version breaks the parsing code; losing track of a revision ID breaks the claim about what was actually measured.

`"Herbert Simon"`, `"Alan Turing"`, and `"Not A Real Laureate Page"`, then

```{python}
def resolve_titles(titles):
    rows = []
    params = {
        "action": "query",
        "format": "json",
        "prop": "info",
        "redirects": 1,
        "titles": "|".join(titles)
    }
    data = polite_get(base_url, params).json()

    followed = {}
    for kind in ("normalized", "redirects"):
        for step in data["query"].get(kind, []):
            followed[step["from"]] = step["to"]

    pages = {p["title"]: p for p in data["query"]["pages"].values()}
    for title in titles:
        final = title
        while final in followed:
            final = followed[final]
        rows.append({
            "doc_id": title,
            "resolved": final,
            "page_id": pages.get(final, {}).get("pageid")
        })
    return pl.DataFrame(rows)

titles = [
    "donald knuth",
    "Edsger_W_Dijkstra",
    "Herbert Simon",
    "Alan Turing",
    "Not A Real Laureate Page"
]
resolved = resolve_titles(titles)

resolved.with_columns(
    in_laureates = c.page_id.is_in(laureates["page_id"].to_list())
)
```

**Answer**: `"donald knuth"` and `"Edsger_W_Dijkstra"` each take two hops
(a mechanical `normalized` fix for casing or underscores, then a real
`redirects` pointer) and land back on page IDs already in `laureates`,
exactly the corpus members they were casual stand-ins for. The other three
land on real, live, unflagged page IDs that are not a typo or a missing
title, and none of them belong in `laureates`, for three different
reasons. `"Not A Real Laureate Page"` has a null `page_id`, the familiar
missing case. `"Herbert Simon"` (dropping the "A.") resolves cleanly to
page ID 35817808, a short disambiguation page listing an economist and an
unrelated real-estate developer, the same silent substitution trap
notebook35 hit with John McCarthy and David Patterson, just triggered here
by an abbreviated name instead of a shared one. `"Alan Turing"` is the most
interesting failure: it resolves to a real, single, entirely unambiguous
article, the person the award is named after, who is not himself a
recipient of it, since he died in 1954 and the award was not established
until 1966. A clean resolution with no redirect, no disambiguation, and no
missing flag can still be the wrong page for a population question, which
is exactly why the `in_laureates` check against the corpus's own spine
table, not the resolution outcome alone, has to be the last word.
