10. `.join_asof()` matches each row to the nearest value in another table
rather than an exact one. Build a table of "resume" moments: every row from
question 5's pause filter, keeping only `id` and `press` (renamed
`pause_press`). Asof-join all of `klog` to it, backward, matched by `id`,
and compute `since_pause`, the milliseconds between each keystroke and the
most recent pause that preceded it. Group by whether `since_pause` is under
1000 (recently resumed) and compare the average `dur` and count in each
group, including the group where no pause has happened yet at all.

```{python}
resumes = (
    pauses
    .select(c.id, pause_press = c.press)
    .sort(c.id, c.pause_press)
)

with_resume = (
    klog
    .sort(c.id, c.press)
    .join_asof(resumes, left_on="press", right_on="pause_press", by="id", strategy="backward")
    .with_columns(
        since_pause = (c.press - c.pause_press).dt.total_milliseconds()
    )
)

(
    with_resume
    .with_columns(
        just_resumed = c.since_pause < 1000
    )
    .group_by(c.just_resumed)
    .agg(
        n = pl.len(),
        avg_dur = c.dur.mean().round(1)
    )
)
```

**Answer**: `just_resumed` comes back with three groups, not two, since
`since_pause` is `null` for the 42,678 keystrokes typed before anyone's
first pause. Those `null` rows hold the shortest average, 74.2 ms, keys
typed within a second of resuming hold the longest, 80.8 ms, and everything
typed a full second or more after resuming sits in between at 76.9 ms. That
ordering is suggestive of a real restart cost, but read it cautiously:
question 2 already showed hold times climbing as a session goes on, and by
the time most people have hit even one 2-second pause they are already well
past their first minute, so part of this gap could be question 2's slowdown
showing up again rather than a genuine cost of resuming. Separating the two
would need controlling for elapsed time directly, which is more than this
question asks for.
