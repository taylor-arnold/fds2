9.1 Setup

Load the modules and datasets needed for the chapter.

9.2 Introduction

The previous chapter focused on combining information stored across multiple tables. This chapter turns to a different kind of transformation: reshaping a single table into a different layout without adding or removing information. We will see that the same set of values can be arranged in several different ways, and that picking the right arrangement, rather than the right values, is often what determines whether an analysis is easy or painful.

We start by returning to the unit of observation, the idea that anchors every decision about how a table should be shaped. From there we introduce tidy data, a short set of rules that tells us when a table’s shape matches its unit of observation. The rest of the chapter puts those rules to work: pivoting and unpivoting to move between wide and long layouts, reshaping data so that it plays well with plotnine, and exploding list columns so that multiple values packed into one cell get a row each. We close by noting the one restructuring operation that does not fit this chapter’s theme, because it destroys information rather than merely rearranging it.

9.3 Units of Observation

When designing a way to store tabular data, we often must decide whether to favor many rows and few columns or many columns and few rows. A related decision, particularly when choosing a format with more columns, is what information to store in columns versus rows. In some cases the answers are clear and obvious. In others, there are trade-offs about which types of analyses will be easiest or hardest to perform with the data. Let’s work through an example.

Assume that you are a biologist tracking the heights of 100 plants over the next 14 days. There are several different ways that this data could be collected. We could have each day’s measurements in a column and each plant as a row. We could have each plant as a column and each day as a row. Or we could have a row for each combination of plant and day, with 100*14 = 1,400 rows and three columns (plant ID, day ID, and height).

A core data-science concept will help us distinguish the three cases above. A unit of observation is the entity about which data are collected and recorded in a dataset. In other words, it is the “who” or “what” that each row of the table represents. In the example with one row for each plant and one column for each day, the unit of observation is a plant. In the example with one row for each day and one column for each plant, the unit of observation is a day. In the final example, the one with 1,400 rows, we call the unit of observation plant × day (read “plant crossed with day”), which means each row corresponds to a unique combination of one plant and one day. The key takeaway is that once we have picked the unit of observation, there is only a single way to construct our data table.

Note that the concept of a unit of observation is closely linked to that of a primary key. By definition, a primary key must uniquely identify the “what” of the unit of observation. For example, a primary key for a table whose unit of observation is a plant must uniquely identify each plant. Likewise, a table with a plant × day unit of observation needs a key that uniquely identifies both the plant and the day.

9.4 Tidy Data

The plant example makes a point that is easy to miss: none of the three layouts is wrong. Each one is a complete, valid way to record the same 1,400 measurements. What differs is which unit of observation each layout is built around, and that difference has consequences for how easy the data is to work with.

Hadley Wickham’s “Tidy Data” gives a name to the layout that matches a chosen unit of observation, and a framework for recognizing it [1]. A dataset is tidy when three conditions hold:

  • every variable is a single column,
  • every observation is a single row,
  • every table holds a single unit of observation.

The second and third rules restate what we already know from the previous section: rows correspond to a unit of observation, and a table should stick to one such unit. The first rule is the new piece. It says that a quantity we might want to filter on, group by, or plot should live in its own column, not be smuggled into a column name, split across several columns, or bundled together with another quantity in the same cell.

The plant-by-day table with 1,400 rows is tidy for the plant × day unit of observation: plant, day, and height each get their own column, and each row is one measurement of one plant on one day. The plant-by-column table is not tidy for that same unit. Day is no longer a variable stored in a column; it has been scattered across fourteen column names, and a single observation (one plant’s full trajectory) is spread across fourteen cells instead of appearing once.

This is the detail that makes tidiness easy to misjudge: it is a property of a table relative to a unit of observation, not a property of the values themselves. The wide plant-by-day table is a perfectly tidy table for the unit of observation plant, if what you want is one row summarizing each plant’s height over the two-week window. It is only untidy once you decide the unit of observation you actually need is plant × day.

We will see this play out concretely later in the chapter with the cellphone dataset, shown below. Its unit of observation is year × country, and in this shape it is tidy: year, iso, and cell each get a column, and each row is one country’s rate in one year.

9.5 Pivot and Unpivot

Sometimes the unit of observation in which a data table is stored does not match the unit needed for our analysis. If the stored format uses a compound unit of observation, we can simplify it by using an aggregation to remove one or more components. For example, we could group a plant × day table by plant ID and calculate summary statistics (such as the maximum height over all days) to generate a dataset with plant as the unit of observation. While aggregations are useful, by design they remove information present in the original dataset. Other times, we want to restructure the unit of observation without losing or adding any information. We simply need the data in a different shape for a particular analysis or visualization. To do this, we can use one of two additional data manipulation functions.

We could have stored the cellphone dataset in a format where the unit of observation was year and each country had its own column. What if we want to work with the data in that format? For example, if we wanted to compute the difference in cellphone ownership between the United States and Canada, we would need the data in this format. We can convert our dataset using the .pivot() method, which requires indicating which column will be used to index the output (this becomes the unit of observation), which column will be used to create the new columns, and which column contains the values that will fill the table.

cell_wide = (
    cellphone
    .pivot(index=c.year, on=c.iso, values=c.cell)

It is tidy for the unit of observation year, and it is exactly the layout we would want if we were computing something like the gap between two specific countries. It is not tidy for the unit of observation year × country, since country has moved out of the data and into the column names.

It is possible that some combinations of the new rows and columns do not exist in the original DataFrame. When this happens, Python will fill in null values for the missing combinations. Sometimes, particularly if these are counts that we are looking at, we may want to convert all of these missing values to zeros (or another default). To do that, we can call the method .fill_null(0). The value zero can be changed to any other value that makes sense. This can also be used on individual columns as needed in other analyses.

WarningPivot and unpivot syntax

The current version of Polars on Google Colab does not allow using the c. notation when specifying columns as we do with other Polars methods. Instead, you must specify the column as a string, similar to how we do it in plotnine graphics. This has already been changed in the development version of Polars for pivot and will likely change for unpivot in the near future. Check this callout against the version of Polars pinned for the course before each printing of the text; it is describing a moving target and may already be out of date by the time you read it. For now, remember that both pivot and unpivot require a different way of specifying columns compared to other Polars methods.

The .pivot() and .unpivot() methods are powerful tools for data manipulation and constitute the final major methods we need for manipulating tabular datasets. Although they may at first glance seem straightforward (and should be, at least in theory), care is needed when using them to reshape data, because they can easily introduce subtle errors. The best practice is to decide on the unit of observation up front and structure your data accordingly. Use these reshaping functions when necessary, but avoid unnecessary back-and-forth transformations that can introduce mistakes.

9.6 Pivoting for plotnine

The tidy data framework was built with a particular use case in mind: it makes plotting predictable. plotnine expects long data, and once you know why, several things about earlier chapters start to make more sense.

Try mapping a plotnine aesthetic to country using cell_wide. There is no iso column to map, only a separate column for each country, so there is no sensible way to write color=c.iso. Nothing in cell_wide tells plotnine that usa and can and every other column name are all values of the same underlying variable.

(
    ggplot(cell_wide, aes(x=c.year, y=c.usa))
    .geom_line()
)

This code runs, but it only gets you a single country’s line, hardcoded by column name, and there is no way to extend it to “one line per country” without writing a separate geom_line() call for every column. Unpivoting fixes this by putting country back into the data as a variable:

cell_long = (
    cell_wide
    .unpivot(index=c.year, variable_name=c.iso, value_name=c.cell)
)

(
    cell_long
    .pipe(ggplot, aes(x=c.year, y=c.cell, color=c.iso))
    .geom_line()
)

With iso as a column, color="iso" is enough to tell plotnine to draw one line per country. This is also the answer to a question you may have had back when we first plotted cellphone by country: that plot worked on the first try, with no reshaping, because cellphone was already stored long, with one row per year × country combination. The general rule is this: before reaching for ggplot(), check whether the variable you want to map to an aesthetic (color, shape, facet, and so on) has its own column. If it does not, unpivot first.

9.7 Expand List Columns

A final type of transformation arises when multiple elements occupy a single cell in a table. This usually occurs only after a data-cleaning step we’ve performed, rather than as raw input from a CSV file. For example, the function str.split can split the text in a DataFrame column into parts. We can use this to split the multiple language codes found in the lang column of our data.

(
    country
    .select(c.full_name, c.lang)
    .with_columns(
        lang_codes = c.lang.str.split("|")

)

9.8 Round Trips Versus Aggregation

It is worth being precise about what pivoting and unpivoting do and do not do. They are round trips: pivot a long table to wide, unpivot it back, and (aside from row and column ordering) you have the original table. No values are created or destroyed, only rearranged. Aggregation is not like this. Grouping a plant × day table down to a plant table by taking the maximum height, for instance, throws away every day’s measurement except the one that set the maximum, and there is no operation that gets those measurements back. Pivoting and unpivoting change a table’s shape; aggregation changes its unit of observation, and it does so in one direction only. Keep the distinction in mind, since it is exactly what motivates the “prefer longer” advice for storing data in the first place: a long table can always be pivoted into whatever wide shape an analysis needs, but a table that has already been aggregated cannot be un-aggregated back into it.
