13  Temporal Data

13.1 Setup

Load all of the modules and datasets needed for the chapter.

import polars as pl
from plotnine import ggplot, aes, element_text
from polars import col as c

import funs

meta = pl.read_parquet("data/wiki/uk/meta.parquet")
page_views = pl.read_parquet("data/wiki/uk/pageviews.parquet")
page_revisions = pl.read_parquet("data/wiki/uk/revisions.parquet")

13.2 Introduction

This chapter is about data with a temporal component: any dataset with one or more variables recording dates or times, and more broadly any dataset whose rows have a meaningful order. We begin with data that carries explicit dates and times, since that is where the special data types and functions live. The later sections widen the scope to ordered data generally, which is why the tools there apply well beyond time series.

The data comes from the corpus of 75 British and Irish authors that we assembled at the end of Chapter 12. Two of its tables are temporal. One records how many people viewed each author’s Wikipedia page on each day of 2025. The other records the most recent 500 edits made to each author’s page, with a timestamp, an editor, the size of the article after the edit, and the editor’s summary comment. A third table gives each author’s birth and death years, which we will need at the end of the chapter.

A word about what is new here and what is not. Two of this chapter’s central tools are ones you have already met. Window functions were the subject of Chapter 6, where .shift() and .over() were introduced on the cellphone data, and conditional joins appeared in Chapter 8, where .join_asof() and .join_where() matched countries on inequalities rather than exact keys. Both come back in this chapter, and both sections say so plainly when they do. This is deliberate rather than accidental. Those tools were introduced on data that had an order but no clock, and time is where they do their real work, so it is worth walking through them a second time on data that actually has dates in it. Where a section is retracing ground, it opens by naming the chapter that covered it and summarizing what you should already know, then moves on to the part that is genuinely new. If a reprise feels too fast, the original treatment is the place to go back to.

The material that is new to this chapter is the temporal type system itself: dates, datetimes, time zones, truncation, and durations, along with the two window and join operations that only make sense once a dataset has real timestamps in it.

The time variables are given the way Chapter 10 recommended for recording dates, with individual columns for year, month, and day. Because all three are stored as numbers, the data can be put in chronological order with .sort() on year, then month, then day. It already is.

How could we show the change in page views over time for a single author? One approach is to add a numeric column running down the dataset using a row number. Below is an example of the code to create a line plot using this approach:

(
    page_views
    .filter(c.doc_id == "Geoffrey Chaucer")
    .with_row_index("row_number")
    .pipe(ggplot, aes(c.row_number, c.views))
    .geom_line(color="red")
    .labs(
        title="Page Views for Geoffrey Chaucer (by Day Number)",
        x="Day Number",
        y="Views"
    )
)

This is not a bad place to begin. The x-axis is the day number, and in many applications that is exactly what we need. Chaucer’s page holds a fairly stable level of around fourteen hundred views a day, with a good deal of short-term noise and a few pronounced spikes. But notice how hard it is to say anything about when those spikes happened. The tallest one sits somewhere around day forty-five, which is not a fact anybody can use. There is also a hidden assumption in the plot: the row number only tracks the date because the data happens to be sampled uniformly, one row per day, with nothing missing. Sort the data differently, or lose a few days, and the axis quietly starts lying.

Another way to work with dates is to convert them to a fractional year. Here the months and days are folded into the year as a fraction:

\[ year_{frac} = year + \frac{month - 1}{12} + \frac{day - 1}{12 \cdot 31}\]

We subtract one from the month and day so that, for example, 1 July 2020 (halfway through the year) comes out as 2020.5. We could make this more exact by accounting for months shorter than 31 days, but as a first pass it works well.

(
    page_views
    .filter(c.doc_id == "Geoffrey Chaucer")
    .with_columns(
        year_frac = c.year + (c.month - 1) / 12 + (c.day - 1) / (12 * 31)
    )
    .pipe(ggplot, aes(c.year_frac, c.views))
    .geom_line(color="red")
    .labs(
        title="Page Views for Geoffrey Chaucer (Fractional Year)",
        x="Fractional Year",
        y="Views"
    )
)

This improves on the first plot in a few ways. We can roughly read off what date each point corresponds to, and the code works whether or not the data is sorted, evenly spaced, or complete. The cost is an axis that needs explaining to anyone who reads the figure. The same approach extends to times. If we also knew the hour of each observation, the formula would become:

\[ year_{frac} = year + \frac{month - 1}{12} + \frac{day - 1}{12 \cdot 31} + \frac{hour - 1}{24 \cdot 12 \cdot 31}\]

And if we only cared about time elapsed since some event, such as the start of an experiment, we could take the difference from a fixed fractional year.

Fractional times are genuinely useful. They can represent an arbitrarily precise moment as an ordinary number, which means they slot into models and functions that know nothing about dates and require no special handling. They are a good way to explore data quickly. But when we want axis labels a reader can understand, or want to ask questions like “what day of the week was this,” we need a type that knows it is a date.

13.4 Date Objects

Most of the variables we have worked with so far are strings or numbers. Dates sit somewhere in between. They are like numbers in that they have a natural order, differences between them are meaningful, and they can be placed on a continuous scale. But they also have structure that numbers do not: a date belongs to a month, falls on a weekday, and sits some irregular number of days from the next date. Extracting that structure requires a dedicated data type. In Polars, pl.date() constructs one from its components.

chaucer = (
    page_views
    .filter(c.doc_id == "Geoffrey Chaucer")
    .with_columns(
        date = pl.date(c.year, c.month, c.day)
    )
)

13.5 Datetime Objects

The page view data records a date and nothing finer. Often we know more precisely when something happened, down to the hour, minute, or second. We use the term datetime for an object that stores a precise moment, which requires both a date and a time of day. Dates have a natural precision of one day; datetimes do not, and the useful granularity depends entirely on the data. Some sources give only the hour, while others, such as broadcast logs, record milliseconds. Polars stores datetimes with microsecond precision by default, which is finer than anything we will need here.

The revision histories carry exactly this kind of timestamp. Recall from Chapter 12 that we deliberately wrote them out as strings, in the YYYY-MM-DD HH:MM:SS format that Chapter 10 recommends, rather than parsing them at collection time. Turning them into datetime objects is the first thing this chapter does with them.

page_revisions = (

    .with_columns(

The timestamp column now has the type Datetime, giving the second at which each edit was saved. Because the string was written in the ISO-8601 format, .str.to_datetime() parses it without being told the layout. A format string can be passed if the data uses some other convention, and often must be, since a date like 03/04/2025 is ambiguous in a way that 2025-04-03 is not. Each row also carries the editor’s username (anyone can edit a Wikipedia page), the size in bytes of the article after the edit, and a short comment describing the change. Plotting size against time shows the article growing, and shows the moments when large sections were added or removed.

selected_authors = ["Geoffrey Chaucer", "Emily Brontë"]

(
    page_revisions
    .filter(c.doc_id.is_in(selected_authors))
    .pipe(ggplot, aes(c.timestamp, c.size, color=c.doc_id))
    .geom_line()

    .labs(

        x="Date",
        y="Page Size (bytes)",

    )
)

The two lines cover very different stretches of time, and the reason is worth pausing on. We collected the 500 most recent edits for every page, not every edit ever made, so the window we observe is not a fixed span of years but a fixed number of events. Emily Brontë’s page is edited often, so her last 500 edits reach back only to late 2024. Chaucer’s page moves more slowly, so the same 500 edits carry us back to 2021. The observation window is different for every author, and any comparison that forgets this will end up comparing a busy year and a quiet half-decade as though they were the same thing.

We can label the axis with scale_x_datetime(), which works like scale_x_date() but understands times as well as dates.

(
    page_revisions

    .pipe(ggplot, aes(c.timestamp, c.size, color=c.doc_id))
    .geom_line()
    .scale_color_manual(values=["red", "blue"])
    .scale_x_datetime(date_breaks="1 year", date_labels="%Y")
    .theme(axis_text_x=element_text(angle=90, hjust=1))
    .labs(

        x="Date",

)

13.6 Language and Time Zones

So far our summaries of dates and times have been numeric. Notice that the weekday example above gave us the numbers 1 through 7 rather than names. Polars can produce string names for weekdays and months, but the output depends on the locale configured on the machine running the code, which makes it a poor thing to build an analysis on. The numeric components below are portable, and they are what most analyses want anyway.

(
    page_revisions
    .head(10)
    .with_columns(
        weekday_num = c.timestamp.dt.weekday(),
        month_num = c.timestamp.dt.month(),
        hour = c.timestamp.dt.hour(),
        date_only = c.timestamp.dt.date()
    )
    .select(c.timestamp, c.weekday_num, c.month_num, c.hour, c.date_only)

The other regional problem is time zones. The concept seems simple and the practice is anything but; an impressive share of the bugs in the history of computing trace back to some edge case in time zone handling.

A datetime in Polars is either naive, carrying no time zone information at all, or aware, tagged with a specific zone. This distinction matters because the two are not interchangeable, and two different methods move between them. The .dt.replace_time_zone() method attaches a zone to a naive datetime, asserting what the clock reading meant without changing the reading itself. The .dt.convert_time_zone() method takes an aware datetime and expresses the same instant in a different zone, which does change the reading. Attaching is a claim about what the data always meant; converting is an arithmetic operation on a known instant.

Our timestamps came out of the parse naive, because the string we stored carried no zone. But we know from Chapter 12 that Wikipedia reports all revision timestamps in UTC, as most technical sources with a global audience do. So the correct first move is to attach UTC, and only then convert.

(
    page_revisions
    .head(10)
    .with_columns(
        utc = c.timestamp.dt.replace_time_zone("UTC")
    )
    .with_columns(
        nyc = c.utc.dt.convert_time_zone("America/New_York")
    )
    .select(c.utc, c.nyc)
)

It is tempting to skip the first step and call .dt.convert_time_zone() on the naive column directly. Polars will let you, and on this data it even gives the right answer, which is precisely what makes it dangerous. Converting a naive datetime silently assumes it was UTC to begin with. That assumption happens to hold for us, so nothing breaks and nothing warns. Feed the same code a column of naive local timestamps from some other source and it will shift every one of them by the offset from UTC, quietly and without complaint. Writing .dt.replace_time_zone("UTC") first costs one line and turns a hidden assumption into a statement in the code, where a reader can check it.

The mirror-image mistake is calling .dt.replace_time_zone() on a column that is already aware. That overwrites the zone label while leaving the clock reading untouched, which moves the instant by however many hours separate the two zones. Attaching a zone and converting between zones look similar and do opposite things, and confusing them is the most common way to corrupt a timestamp.

With the zone attached, we can present the data in the local terms of whatever audience we have. The code below counts edits by the hour of the day in New York.

hourly_edits = (
    page_revisions
    .with_columns(
        hour_nyc = (
            c.timestamp
            .dt.replace_time_zone("UTC")
            .dt.convert_time_zone("America/New_York")
            .dt.hour()
        )
    )
    .group_by(c.hour_nyc)
    .agg(
        count = pl.len()
    )
    .sort(c.hour_nyc)
)

(
    hourly_edits
    .pipe(ggplot, aes(c.hour_nyc, c.count))
    .geom_col()
    .labs(
        title="Wikipedia Edits by Hour (New York Time)",
        x="Hour of Day",
        y="Number of Edits"
    )
)

Plenty of editors of course live in London, Los Angeles, or Mumbai rather than New York, and the choice of zone is a rhetorical one as much as a technical one. But most readers find it far easier to reason about a distribution stated in a clock they use than one stated in UTC.

13.7 Truncating Dates

We have seen how to build date and datetime objects and how to extract their components. A related operation is reducing a timestamp’s precision, which is more useful than it might sound. Our revisions are recorded to the second, and that is finer than almost any question we want to ask of them. Grouping by an exact second would put nearly every edit in a group of its own.

The .dt.truncate() method rounds a date or datetime down to a given duration. Setting the duration to "1h" discards the minutes and seconds; "1d" discards the time entirely.

(
    page_revisions
    .head(10)
    .with_columns(
        ts_hour = c.timestamp.dt.truncate("1h"),
        ts_day = c.timestamp.dt.truncate("1d")
    )
    .select(c.timestamp, c.ts_hour, c.ts_day)
)

13.8 Window Functions Revisited

Window functions were the subject of Chapter 6, so this section is a return visit rather than an introduction. The two ideas you should carry in from that chapter are these. First, a window function computes a value for every row while looking at other rows, which is what separates it from an aggregation: .agg() collapses a group to one row, while .over() writes the group’s answer onto every row of the group and keeps them all. Second, the functions that look up and down a column, .shift() chief among them, depend entirely on the order of the rows, which means they belong immediately after a .sort(), and they need .over() to stop them running across the boundary between one group and the next. In Chapter 6 that was the cellphone data, where a year-over-year change computed without .over(c.iso) would have subtracted the last year of one country from the first year of the next.

That is the whole toolkit, and it transfers here unchanged. What changes is that our rows are now separated by real, irregular stretches of time, which opens up questions the cellphone data could not ask.

Take the page revisions. We have the size of each article after each edit, and what we would like is the size of each edit: how many bytes it added or removed. That value is not in the data, but it is the difference between one row and the row before it, within the same page. The pattern is exactly the one from Chapter 6.

The avg_gap column is itself a duration, which reads well but does not plot or compare across units. For that we convert with one of the .dt.total_*() methods, of which .dt.total_hours() above is one; the family also includes .dt.total_days(), .dt.total_minutes(), and .dt.total_seconds().

Choose that unit with some care, because these methods truncate toward zero rather than rounding. The unit needs to be finer than the intervals being measured.

Remember too the caveat from earlier. Because we hold the 500 most recent edits rather than the full history, the authors at the top of this table are the ones whose observation window is shortest. The ranking describes editing intensity in the recent past, not across the whole life of the encyclopedia.

13.10 Rolling Windows in Time

There is one more window function to meet, and it is the one that could not have been taught in Chapter 6. That chapter introduced .rolling_mean(), which smooths a series by averaging each row with the rows near it, and defines “near” as a count of rows: window_size=5 means this row and the four above it. It also flagged, in its closing section, that Polars offers time-aware versions of these functions, and left them for later. This is later.

The distinction matters enormously here, and the revision data shows why. A row-count window assumes that consecutive rows are equally spaced, which is exactly what the cellphone data guaranteed and what our revisions violate. Two consecutive edits to Chaucer’s page might be forty seconds apart or eight months apart. A five-row window over that series averages together whatever five edits happen to be adjacent, spanning a day in one place and a year in another. The window is measured in the wrong units.

What we want is a window defined by a span of time: average the article size over the last thirty days, however many edits that happens to include. The .rolling_mean_by() method does this, taking the column holding the timestamps as by and the span as window_size, written as a duration string.

(
    page_revisions
    .filter(c.doc_id == "Geoffrey Chaucer")
    .sort(c.timestamp)
    .with_columns(
        size_smooth = c.size.rolling_mean_by(by=c.timestamp, window_size="30d")
    )
    .pipe(ggplot, aes(c.timestamp, c.size))
    .geom_line(color="grey")
    .geom_line(aes(y=c.size_smooth), color="red")
    .labs(
        title="Chaucer Page Size, Raw and 30-Day Rolling Mean",
        x="Date",
        y="Page Size (bytes)"
    )
)

The grey line is the raw size after every edit, jagged with vandalism and its reversal. The red line averages over a thirty-day window and shows the article’s actual trajectory. Note that no .over() appears here only because we filtered to a single author first; computing this across the whole table would need .over(c.doc_id) for the same reason every other ordered function does.

Polars provides the whole _by family on the same principle, including .rolling_sum_by(), .rolling_max_by(), and .rolling_std_by(). Whenever rows are unevenly spaced in time, these are almost always what you want, and the row-count versions are almost always a subtle mistake.

13.11 Joining on Time

We close with joins, which we last saw in Chapter 8. Recall the distinction that chapter drew. An ordinary join matches rows where a key in one table equals a key in another, which covers most of what anybody needs. Conditional joins relax the equality. The .join_where() method keeps pairs of rows satisfying any condition you can write as a filter expression, and .join_asof() matches each row to the nearest value in the other table rather than an exact one, which is much faster but requires both tables to be sorted on the join key.

Chapter 8 demonstrated both on the country data, matching African countries to Asian ones by Gini coefficient. It also noted, in passing, that .join_asof() was invented for time series and got its name from asking what a value was as of a particular moment. That is the case we can finally make properly.

Here is the problem. We have two tables about the same authors, measured on different clocks. Page views are recorded once per day. Revisions are recorded to the second. Suppose we want to know how many people were reading a page on the day it was edited, perhaps to ask whether edits follow attention. There is no exact key to join on, because no revision happened at midnight precisely, which is the only timestamp the page view table contains.

An asof join is built for exactly this. We give each daily view count a datetime, sort both tables, and ask Polars to match each revision backward to the most recent view record at or before it.

views_daily = (
    page_views
    .select(
        c.doc_id,
        moment = pl.date(c.year, c.month, c.day).cast(pl.Datetime),
        views = c.views
    )
    .sort(c.moment)
)

edits_2025 = (
    page_revisions
    .filter(c.timestamp.dt.year() == 2025)
    .sort(c.timestamp)
)

edits_with_views = (
    edits_2025
    .join_asof(
        views_daily,
        left_on="timestamp",
        right_on="moment",
        by="doc_id",
        strategy="backward"
    )
)

This table is a network, with authors as nodes and shared lifetimes as edges, and it is exactly the sort of structure that Chapter 16 takes up.

13.12 Summary

This chapter added a type system for time to the tools we already had. Dates and datetimes are distinct from the numbers that encode them, and constructing them with pl.date() or .str.to_datetime() unlocks the .dt namespace, sensible axis labels, and comparisons that respect the calendar. Time zones divide into naive and aware datetimes, and the difference between attaching a zone and converting between zones is the difference between a claim and a calculation. Truncation reduces precision on purpose, turning a stream of timestamps into bins that can be grouped. Durations arise from subtracting one moment from another and convert into whatever unit the question calls for.

The chapter also revisited two sets of tools deliberately. The window functions of Chapter 6 came back on data with real timestamps, where .shift() and .over() behave exactly as before but let us measure the size of an edit, detect a reversion, and time the gap between events. That revisit also paid off two things Chapter 6 left open: .diff() as the shortcut for a differenced shift, and the _by family of rolling functions, which define a window as a span of time rather than a count of rows and are the right tool whenever observations are unevenly spaced. The conditional joins of Chapter 8 came back too, and .join_asof() finally got the time series problem it was designed for.
